Pipeline Operations Manager

EPIC-3 — Infrastructure Lifecycle Management
Design Document | V1 (dev) | April 2, 2026 | Build 389

Overview

The Pipeline Operations Manager is an AI agent that manages infrastructure lifecycle for ChatHealthy.ai data pipelines. It manages clusters, not data. It reads errors, not records. It escalates to the right person, not fixes code.

The agent uses tools. Tools don't think. The agent thinks.

Ownership Boundary — Ops Manager vs Dev Manager

The Ops Manager and Dev Manager are separate concerns with a strict boundary. The Ops Manager never reads pipeline code. The Dev Manager never touches infrastructure. They communicate through one handoff point: an evidence package.

Evidence Package — The Only Handoff

error_messageFull exception textExact error

The Dev Manager receives this package and nothing else. It uses the evidence to find the relevant code, diagnose the bug, and either fix it or escalate to human. The recent_events field uses a stable JSON schema and is capped at 10 entries to prevent log explosion.

Allowed Repair Actions (config-driven allowlist)

GPT-4.1-mini may only select from this list. It cannot invent actions. Stored in Brain config.

ActionWhat It DoesRisk
restart_clusterResume paused cluster via Atlas APILow — idempotent
reconnectDrop and re-establish connection poolLow — existing queries may fail
scale_upIncrease cluster tier (within configured cap)Medium — cost increase
flush_connection_poolClear stale connectionsLow — brief disruption

Repair Loop Guardrail — Evidence Must Change

After each repair attempt, the Ops Manager compares the new evidence (cluster state + error signature) to the previous attempt. If evidence is unchanged, remaining retries are skipped and the system escalates immediately. This prevents churn where the same failing repair is repeated M times with no progress.

After gathering evidence, GPT-4.1-mini classifies:

VerdictEvidence RequiredNext Step
InfrastructureCluster down, connection failed, timeoutSelf-repair → if fails → Escalate human
Pipeline LogicSchema mismatch, data format error, business rule violationSend to Dev Manager with evidence
Can't DetermineMax iterations reachedEscalate human directly

Error Classification — Rule-Based Precheck

Before calling GPT-4.1-mini, the Ops Manager applies deterministic rules. If a rule matches, classification is instant and free. LLM triage is only invoked when rules are inconclusive.

SignalAuto-ClassificationRationale
ConnectionFailure, ServerSelectionTimeoutErrorINFRASTRUCTURECluster unreachable — always infra
AuthenticationFailed, TLS handshake errorINFRASTRUCTURECredential or network layer
Cluster state = PAUSED or REPAIRINGINFRASTRUCTURECluster not ready — always infra
KeyError, TypeError, ValidationError in pipeline codePIPELINESchema or data format — always code
IndexError, ValueError with traceback in Code/DataPipelines/PIPELINEETL logic error
None of the above→ LLM TRIAGEAmbiguous — send to GPT-4.1-mini

UAT Test Scenarios — Failure Path Validation

These scenarios are executed during UAT against real systems. The Ops Manager should handle all of these transparently — human gets notified, the right path fires, no human intervention needed unless escalated.

Infrastructure Failure Path (Ops Manager owns)

#ScenarioHow to TriggerExpected Behavior
UAT-I1 Database killed mid-job Pause the Atlas cluster manually while a pipeline job is running Rule-based precheck classifies as INFRASTRUCTURE. Repair loop attempts restart_cluster. Verifies via read_logs + status. Notifies human on detection and resolution. Audit trail logged.
UAT-I2 Connection timeout Rotate Atlas credentials (invalidate ATLAS_PRIVATE_KEY) during a running job ServerSelectionTimeoutError or AuthenticationFailed → rule-based INFRASTRUCTURE. Repair loop attempts reconnect. If credentials invalid, evidence unchanged → early escalation to human (evidence-must-change guardrail).
UAT-I3 Cluster stuck in non-IDLE state Trigger a resize operation then immediately start a job before resize completes Timer detects cluster stuck (non-IDLE > 30 min). Notifies human with cooldown (30 min per cluster). No task execution.
UAT-I4 Orphan cluster — running with zero reservations Kill the Azure Function mid-job (before release() runs in finally block) Timer detects cluster running with zero reservations. Pauses cluster. Notifies human.
UAT-I5 Repair exhaustion — M attempts fail Pause cluster + block Atlas API (e.g., firewall rule) so restart_cluster fails repeatedly Repair loop runs M times. Evidence unchanged after each attempt → escalates to human before M. Audit trail shows all attempts + early exit reason.

ETL Code / Logic Failure Path (Dev Manager owns)

#ScenarioHow to TriggerExpected Behavior
UAT-L1 Collection missing Drop a target collection before running a pipeline that writes to it Pipeline throws KeyError or collection-not-found. Rule-based precheck classifies as PIPELINE. Evidence package sent to Dev Manager. Dev investigates, finds missing collection, creates it or reports.
UAT-L2 Collection empty (unexpected) Clear all documents from a source collection that a pipeline reads from Pipeline returns zero records or throws on empty cursor. Classified as PIPELINE (business rule violation — expected records, got zero). Evidence package to Dev Manager.
UAT-L3 Schema mismatch Add a required field to the expected schema but don't update the source data ValidationError or KeyError in transform code → rule-based PIPELINE. Evidence package to Dev Manager with traceback showing Code/DataPipelines/ origin.
UAT-L4 Recursive loop / infinite retry Inject a deliberate recursive call or infinite retry loop in a pipeline step Job exceeds expected_duration → timer flags as overdue. Notifies human (cooldown: 60 min per job). If job eventually errors out, triage classifies based on error type.
UAT-L5 Job finishes too quickly (suspiciously fast) Run a pipeline against an empty or near-empty dataset without pre-validation Job completes in < expected_duration * threshold (e.g., <10% of expected). Ops Manager flags as WARNING and notifies Dev Manager to verify output. Not an error — a signal that something may be wrong.

Ambiguous / Edge Cases

#ScenarioHow to TriggerExpected Behavior
UAT-A1 Error that rules can't classify Throw a custom exception type not in the rule-based precheck list Rules → inconclusive → GPT-4.1-mini triage invoked. LLM classifies with reasoning. Routes to correct path (infra or Dev). If LLM can't determine after N iterations → escalate human.
UAT-A2 Mixed failure — infra + logic Drop a collection AND pause the cluster simultaneously First error classified (likely INFRASTRUCTURE since cluster is down). After infra repair, if pipeline still fails → second triage round classifies as PIPELINE. Both errors handled in sequence, not lost.
UAT-A3 Notification flood Trigger 10+ errors in rapid succession (e.g., batch of failing jobs) Cooldown windows prevent spam. Error detected: immediate (each unique error). Job overdue: 60 min per job. Cluster stuck: 30 min per cluster. Human not buried.
UAT-A4 Concurrent reservation failure Three jobs running, one fails — other two should continue unaffected Failed job's error goes through triage. Surviving jobs keep their reservations. Cluster stays up (array non-empty). Only the failed job is investigated.

Audit Trail Validation

#ScenarioValidation
UAT-AT1 After any UAT-I or UAT-L scenario Query audit collection. Verify: triage record exists with timestamp, classification, source (rule/LLM). Repair records exist for each attempt. Escalation record exists if escalated. No records updated or deleted.
UAT-AT2 After UAT-L5 (too-fast job) Verify warning record logged with job_id, actual_duration, expected_duration, and flag type = "suspiciously_fast".
