Provider Data Load & County Enrichment — Design

March 2026

1. Overview

This document covers the design for the provider data pipeline. For the two-cluster strategy and data source roadmap, see the Architecture Overview — Section 9.

The top-level entry point is FullProviderPipeline — a 5-step Durable Functions orchestrator that runs health check → load → county enrichment → embeddings in sequence. All steps are resumable via the start_step payload parameter (default: 1, full run).

LoadProviderData (Step 2) ingests the NPPES NPI file (~8 GB unzipped, ~8.8 million records — Type 1 individual providers and Type 2 organizational facilities) from CMS into PublicHealthData.providers_staging. The load runs asynchronously via Azure Durable Functions; the Router returns a status URL immediately. Progress is tracked in admin.DataLoadMetadata. A load report is sent via SparkPost email on completion.

SLA: full 8M-record load completes in under 1 hour. First production run (2026-03-19): 8,858,520 records loaded in ~35 minutes with 32 workers on Azure East US 2 + MongoDB Atlas M30. County enrichment and embedding generation run as Steps 3–5 of FullProviderPipeline after load completes.

3b. County Enrichment Flow

County enrichment runs as a separate Durable Function after the provider load completes. The key insight: ~8.8M provider records contain ~40K distinct ZIP codes. Rather than enriching each record individually (8.8M operations), we fan out over ZIP codes and issue one updateMany per ZIP — roughly 220× fewer operations.

Pass 1 data source: HUD USPS ZIP-to-county crosswalk (huduser.gov/portal/datasets/usps_crosswalk.html), published quarterly. Only ZIP codes with res_ratio ≥ 0.98 are resolved in Pass 1 — these are effectively single-county ZIPs. The crosswalk is loaded into PublicHealthData.ZipCountyCrosswalk as a prerequisite before enrichment runs.
Pass 2 / 2b data source: US Census Geocoder API (free, US-only). For providers in split ZIPs (res_ratio < 0.98), the practice address is submitted first (Pass 2). If the practice address is blank, the mailing address is tried instead (Pass 2b). Pass 2 is record-by-record — updateOne per provider.

3c. Multi-Pass Enrichment Strategy

County enrichment follows three escalating tiers — data first, web second, AI last. Each pass runs only against records the previous passes could not resolve. Every record is tagged with its provenance source so the pipeline never re-attempts a record that has already been given a definitive result (success or permanent failure).

PassTierMethodSourceProvenance TagStatus
1DataZIP → county crosswalk · updateMany per ZIPHUD USPS crosswalkcrosswalk_pass1Live
2DataPractice address → Census Geocoder · updateOneUS Census Geocoder APIgeocoder_pass2Live
2bDataMailing address fallback (when practice address is blank) → Census GeocoderUS Census Geocoder APIgeocoder_pass2_billingLive
3WebWeb search by NPIPublic webgeocoder_web_npiPlanned
4WebWeb search by provider name + statePublic webgeocoder_web_namePlanned
5AIAI research agent — reasons across sources to infer countyLLM + webgeocoder_ai_researchPlanned

Records with county.source = geocoder_no_address (both practice and mailing addresses blank after Pass 2b) will proceed directly to the AI pass — web search cannot help without any address signal. Records successfully enriched in any pass are not re-attempted. AI passes are controlled by the do_ai_passes payload flag (default true).

4. Classes

ProviderLoadManager

Durable orchestrator. 9 steps: download ZIP, extract CSV, partition file, drain staging, ensure indexes, write metadata, fan-out up to 200 load workers, reconcile record counts, write report. No timeout. County enrichment is a separate orchestrator triggered by FullProviderPipeline after this orchestrator completes.

ProviderWorker

Reads one byte-range slice of the CSV from blob. Partition boundaries are newline-aligned; workers 2–N defensively discard their first row if it has the wrong field count (partial row from alignment fallback). Parses rows. Skips blank/whitespace-only lines. Normalizes fields: taxonomy codes → taxonomies: [{code, primary, group}]; license numbers + states → licenses: [{state, number}]; practice and mailing addresses → sub-documents with ZIP truncated to 5 digits; other identifiers → arrays. Assigns a synthetic record_id = worker_id × MAX_ROWS_PER_WORKER + local_row_index (MAX_ROWS_PER_WORKER = 1,000,000; max 200 workers × 1,000,000 = 200,000,000 — fits in a 32-bit signed int) — unique across all workers, no coordination needed. NPI is the business key. Stamps each document with load_id (= Durable orchestration instance ID) and a stub county: {fips: null} enriched by CountyEnrichment. Batch-upserts to PublicHealthData.providers_staging using $setOnInsert on (load_id, record_id) — safe to retry; duplicates are silently ignored. Returns observability metrics: rows_processed, num_records, rows_failed, duration_seconds, rows_per_second. Updates its metadata record on completion or failure.

EmbeddingWorker

Reads all documents for a given worker_id that do not yet have an embedding field (idempotent on retry). Builds a text representation from all meaningful fields — name, taxonomies, licenses, addresses, credentials, identifiers — by recursively extracting string values and joining them. Excludes internal housekeeping fields (record_id, load_id, worker_id, county, embedding). Calls OpenAI text-embedding-3-small in batches of 500, stores the returned 1536-dimension float array as embedding on each document via bulk_write.

CountyEnrichmentOrchestrator

Separate Durable Function orchestrator — not part of ProviderLoadManager. Triggered by FullProviderPipeline after ProviderLoadManager completes. 7 steps: get distinct ZIPs, lookup crosswalk, Pass 1 ZIP fan-out, get unenriched providers, Pass 2 address fan-out, reconcile, write report. Prerequisite: PublicHealthData.ZipCountyCrosswalk must be populated by the LoadCrosswalk pipeline before enrichment runs.

ZipEnrichmentWorker (Pass 1)

Accepts a batch of ZIP codes where res_ratio ≥ 0.98. For each ZIP: looks up the county FIPS code from PublicHealthData.ZipCountyCrosswalk and issues one updateMany on providers_staging, adding county_fips and county_name to all matching provider records. Returns count of documents modified.

AddressEnrichmentWorker (Pass 2 + 2b)

Accepts a batch of provider records whose county_fips is still null after Pass 1 (i.e. their ZIP spans multiple counties). For each provider: submits the practice address to the US Census Geocoder API (Pass 2). If the practice address is blank, falls back to the mailing address (Pass 2b). Tags each record with its provenance source: geocoder_pass2, geocoder_pass2_billing, geocoder_failed (geocoder returned no match), or geocoder_no_address (both addresses blank). Issues one updateOne per provider. Returns counts for each outcome.

DiscrepancyReporter

Writes the complete run data (all worker results + reconciliation + summary) as a document to admin.PipelineDiscrepancyReport in MongoDB. Sends a SparkPost email notification with key stats and the load_id for lookup. No blob storage or file generation.

4b. Pipeline Exception Handling Framework

Pattern: Gang of Four — Template Method. PipelineWorkerBase (pipeline_worker_base.py) is an abstract base class that all pipeline row-processing workers extend. It centralises exception handling, error accumulation, and lifecycle control so that worker classes focus purely on data processing logic.

Ownership

The base class owns:

  • The processing loop — pipeline_execute()
  • All exception catching and logging — _pipeline_step() (private template method)
  • The row_errors accumulator — list of {row_key, reason} objects written to the discrepancy report

Control Flow

↓ while _pipeline_has_next()

↓ calls
_pipeline_process()abstract · subclass
success → continue

↓ after loop

↓ success only
_pipeline_build_result()abstract · subclass returns result dict to orchestrator

Abstract Methods — Subclasses Must Implement

MethodContract
_pipeline_has_next()Advance the internal cursor. Return True if a row is ready to process.
_pipeline_process()Process the current row. Raise on error. Must not catch exceptions or log errors — the base class owns the row lifecycle.
_pipeline_row_key()Return the business key of the current row (e.g. NPI number). Called by the base class immediately after an exception is raised.
_pipeline_resume()Advance past the failed row and reset any partial state. Must not raise.
_pipeline_build_result()Build and return the result dict for the orchestrator. Called only on success, after _pipeline_close().

Optional Hooks

MethodDefaultOverride for
_pipeline_open()no-opResource setup — blob streams, DB connections, cursors

Error Policy — fail_on_row_error

Set via fail_on_row_error (bool) in the worker config dict. Default: False.

FalseRow error is logged, appended to row_errors, _pipeline_resume() is called, loop continues. All errors accumulate in the discrepancy report.

8. Assumptions

NPPES CSV — No Embedded Newlines in Quoted Fields
The byte-based partition boundary logic scans for the next \n to align partition boundaries. This is correct only if NPPES data does not contain embedded newlines inside quoted CSV fields. This assumption is empirically consistent with all CMS NPPES dissemination files observed to date. If CMS changes this, partition boundaries will be misaligned, workers will produce rows with wrong field counts (counted as rows_failed), and the reconciliation step will detect the resulting expected_records ≠ inserted_records mismatch. No silent data loss occurs.
Worker Partition Alignment — Newline-Aligned Boundaries
partition_file_fn aligns each boundary to the next \n by scanning a 1024-byte window. In the rare edge case where no \n is found in 1024 bytes (extraordinarily long row), the boundary falls at the nominal byte offset (mid-row). Workers 2–N defensively discard their first row if it has the wrong field count. Complete first rows (normal aligned case) are processed normally.

9. Design Review Notes

The following issues were identified in an external design review and resolved before alpha.

#IssueResolution
1load_id not propagated — workers had no shared run identifier; retries could not be deduped.Orchestrator sets load_id = context.instance_id at start and propagates through all config dicts. Every staging document and metadata record carries load_id.
2No idempotency on retry — InsertOne would duplicate records on orchestration replay.Changed to UpdateOne($setOnInsert) upsert on (load_id, record_id). Safe to replay any number of times.
3No staging index — upsert filter had no supporting index; full collection scans on every row.Added ensure_indexes_activity (Step 5) before workers start. Creates unique compound index on (load_id, record_id). Idempotent.
4CMS scraping had no fallback — runtime failure on CMS page change would abort the entire load.Added NPPES_FALLBACK_URL constant. download_zip_fn catches scrape failures and falls back to last-known URL with a warning log.
5Reconciliation miscounted trailing newline — file ending without \n was undercounted by 1.Track last chunk; if last byte ≠ \n, add 1 to csv_record_count.
6Partition scan window too small (256 bytes) — NPI rows average ~500 bytes; window could miss the newline.Increased scan window to 1024 bytes in partition_file_fn.
7No observability metrics from workers — no way to diagnose throughput or partial failures post-run.ProviderWorker.run() now returns rows_processed, rows_failed, duration_seconds, rows_per_second.
8MAX_ROWS_PER_WORKER implicit — magic number inline in worker code.Extracted to named constant MAX_ROWS_PER_WORKER = 1_000_000 at module level. Max record_id = 200 workers × 1,000,000 = 200,000,000 — fits in a 32-bit signed int.
9Excel report added unnecessary complexity — blob storage, SAS URLs, openpyxl dependency, and a 10K-row cap workaround.Replaced with a MongoDB document written to admin.PipelineDiscrepancyReport. Full run data stored directly; no file generation or blob storage needed for reporting.
10download_zip_fn returned str in original design — auto-discovered version was lost.Returns dict {"zip_path", "version"}. Orchestrator propagates version into config for all downstream activities. (Fixed.)
11max_records was approximate — required sampling rows to estimate a byte offset, adding complexity and a ±5% error margin.Replaced with max_bytes: a direct byte limit applied to the data region after the header. No sampling needed. Used for dev limiting only; production runs omit the parameter.
12Partial failure strategy unspecified — if 1 of 5 workers fails, what happens?All pairs run to completion (no short-circuit). If any worker fails, the orchestrator returns status: "failed". Successful inserts are NOT rolled back — load_id isolates this run. A re-run is safe due to upsert idempotency.
13Staging retained stale records across load runs — old provider data visible alongside new load until collection swap.Added drain_staging_activity (Step 4) to delete all existing documents before workers start. Drain runs only after download + extract + partition succeed, ensuring the file is viable before the old data is removed. Each run's documents are still isolated by load_id for dedup safety.

10. Open Issues

SECURITY — Durable Functions status URL exposes orchestration state without our auth layer
The 202 Accepted response returned to the caller includes a statusQueryGetUri containing an Azure-generated webhook code parameter. Anyone in possession of that URL can poll the run status (load ID, record counts, worker results) without a Bearer token. This is standard Azure Durable Functions behavior but fails enterprise security review — a hospital or insurance company would flag it.

Fix (before enterprise sale): Proxy the status check through our own /api/Router endpoint (authenticated with Bearer token), strip statusQueryGetUri from the 202 response, and store the webhook URL server-side keyed by load_id.

10b. Release Gates — Promoting Data to Production

Pipeline success alone is not sufficient to deploy. Two independent gates must pass before data is promoted from ChatHealthyDataPipelines to ChatHealthyFrontEnd. These are product/business decisions, not engineering decisions — thresholds are defined here, not in the pipeline code.

Gate 1 — SIT (System Integration Testing)

Automated. Run qa_provider_load.py after the pipeline declares done. Exit code 0 = pass; exit code 1 = fail, do not promote.

Any state coverage≥ 85% enrichment per stateInvestigate that state — bad ZIP data, geocoder outage

Gate 2 — UAT (User Acceptance Testing)

Manual. A human (currently: Skip) asks 3–5 real FindCare questions against the chat UI after data is promoted to ChatHealthyFrontEnd and wired into the HuggingFace app. Questions must cover different intents and geographies.

11. Deferred

  • Incremental / delta loads — current design always loads the full NPI file
  • Duplicate NPI detection — some individual providers have multiple NPIs; deduplication requires fuzzy matching on name + taxonomy + credential (see RESEARCH-001 backlog)
  • Multi-location provider data — NPPES captures one practice address per individual NPI; supplementing with payer directory data would surface all billing locations (see RESEARCH-001 backlog)
  • Update NPPES_FALLBACK_URL constant whenever a new monthly file is confirmed working
  • Durable Functions status URL exposes orchestration state without our auth layer — proxy through /api/Router before enterprise sale (see Open Issues)
  • SparkPost sender domain verification — Skip.Snow@mail.chatHealthy.ai must be verified in SparkPost portal
  • Timer Triggers — autonomous pipeline scheduling (monthly NPI load, weekly specialty data) not yet wired
  • Idle monitor re-enable — disabled because it paused the cluster mid-run; requires pipeline lock mechanism
  • Cluster manager agent — dedicated service to start/stop ChatHealthyDataPipelines on schedule, decoupled from the pipeline
