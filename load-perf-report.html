1. Problem

Loading 8.8M NPI provider records into MongoDB took 88.9 minutes — approximately 6× slower than the target SLA. The bottleneck was not hardware saturation. The same workload ran at 460 rows/s per worker in near-isolation but degraded to 170 rows/s under concurrency. The cause was write contention from an upsert pattern that performs a read-before-write on a live unique index, multiplied across 8–12 concurrent workers hitting the same B-tree pages simultaneously.

3. Analysis

The orchestrator dispatched all 32 workers simultaneously, but the Azure Function App's concurrency ceiling held execution to ~8 workers at a time. This produced distinct waves — not by design, but as a platform scheduling artifact.

Workers running in 8-way concurrency averaged 170 rows/s. Workers running in near-isolation averaged 460 rows/s — a 2.7× gap at the same collection size. Critically, the last 4 workers ran against a fully loaded 8.8M-record collection and were the fastest (460–502 rows/s). This rules out index size, collection growth, or WiredTiger cache pressure as root causes.

The bottleneck is write contention on the (load_id, record_id) unique index. UpdateOne($setOnInsert) issues an index lookup before every insert — even when the document is guaranteed not to exist. Under 8-way concurrency, multiple workers compete for overlapping B-tree leaf pages, causing WiredTiger lock contention. The read-before-write overhead scales linearly with concurrency; pure insert does not.

The concurrency ceiling (8 simultaneous workers) also means the fan-out never runs at full 32-worker throughput. At 460 rows/s × 32 workers the projected duration is 10 minutes.

4. Implementation — Optimizations Applied

Commit 5dcb84d, 2026-03-23. All five changes are active in the current run.

  1. InsertOne replacing UpdateOne($setOnInsert)
    • Eliminates read-before-write on every row
    • Staging collection is drained before load — pure insert is safe
    • Duplicate key errors on retry remain the safety net
  2. (load_id, record_id) unique index deferred to post-load
    • Index previously maintained incrementally per batch — every batch triggered B-tree rebalancing
    • Index now built once over the complete dataset after all workers finish
    • ensure_preload_indexes_fn made a no-op; index moved to ensure_postload_indexes_fn
  3. Azure Function App concurrency ceiling raised
    • host.json: maxConcurrentActivityFunctions = 32 (was effectively ~8)
    • Allows all 32 workers to execute simultaneously
    • Only fully effective combined with optimizations 1 and 2 above
  4. Batch size increased 5,000 → 10,000
    • Halves the number of round-trips to MongoDB per worker
    • Larger batches amortize per-call network overhead
  5. MongoClient maxPoolSize raised to 64
    • Prevents connection exhaustion under peak fan-out
