Pipeline Deliverables

RESEARCH-001 Multi-location provider data — payer directory agent April 2026
Status: Scheduled — starts April 7, 2026

NPPES NPI data captures only one practice address per individual provider (Type 1 NPI). If a provider practices at multiple locations, only their primary address appears in FindCare.

The gap: Users searching by location may miss providers who practice nearby but list a different primary address in NPPES.

Solution: An agent that supplements NPPES with payer network directory data. Payer directories are location-level (providers must register each billing location to get paid) and kept more current than NPI.

Data sources:

  • Payer network directories — per-payer, not a single public feed (BCBS, Aetna, UHC, Cigna)
  • CMS Medicare Provider Enrollment data — facilities where a provider has billed Medicare
  • CMS Open Payments — facility-level billing activity
  • CAQH — provider self-reported, credentialing-focused, not public

Why an agent: Each payer directory has a different API/format; an agent handles per-payer variation, retry logic, and reconciliation across sources.

Research Agents

RESEARCH-002 ICD-10 → specialty → provider linking agent Alpha — April 30, 2026
Status: Open Depends on: ICD-10, procedure codes, and CMS Part D data loaded first

Given a patient's diagnosis (ICD-10 code), FindCare needs to identify which provider specialties treat that condition and surface relevant providers along with their prescribing patterns.

The problem: ICD-10 codes and provider taxonomy (specialty) codes are separate classification systems with no built-in mapping between them. A user saying "I have Type 2 diabetes" should find endocrinologists and primary care providers — but that linkage doesn't exist in the raw data.

Proposed approach:

  • Map ICD-10 codes → relevant taxonomy codes — likely LLM-assisted or via CMS/AMA published mappings
  • Use taxonomy codes to filter providers from the NPI dataset
  • Join with CMS Medicare Part D prescriber data to show prescribing patterns for that condition

Data sources:

  • ICD-10-CM codes — LoadICD10 exists in pipeline router
  • Provider taxonomy codes — loaded via LoadSpecialtyData
  • Procedure codes — planned pipeline (Tuesday milestone)
  • CMS Medicare Part D prescriber data — planned pipeline (Tuesday milestone)

Why an agent: The ICD-10 → specialty mapping requires reasoning — one condition may involve multiple specialties and some mappings are nuanced. An LLM agent is better suited than a static lookup table.

INFRA-006 Durable Functions orchestrator versioning April 25, 2026
Status: Open Priority: Pre-Alpha

Deploying new pipeline code while a run is in progress causes a replay failure — confirmed March 22, 2026. A deploy guard is now in place but does not cover all edge cases. Full solution: side-by-side orchestrator versioning so in-flight runs complete against the version they started with while new deploys route to the updated version.

INFRA-008 Pipeline management agent Beta — June 2026
Status: Open

A dedicated agent — separate from the pipeline orchestrator — that monitors all running pipeline instances, detects failures and ghost orchestrations, and takes corrective action autonomously. The orchestrator raises on failure and stops; recovery is this agent's job.

  • Monitors Durable Functions instance status via management REST API (sole owner of that API)
  • Detects stuck Running instances — OOM-killed workers, replay failures
  • Terminates and purges ghost sub-orchestrators automatically
  • Restarts failed pipelines from the correct start_step
  • Reports failures and recovery actions to admin.PipelineOpsLog + alerting

Boundary: This is a system management concern, not a pipeline concern. No cleanup logic lives in the orchestrator. The agent observes external state and acts; the orchestrator executes data work only.

Backlog

INFRA-009 API protection and abuse prevention Post-Alpha High priority when live
Status: Deferred — activates when any compute-backed endpoint is exposed

The public site is static and protected at the Cloudflare edge with basic traffic filtering. That is sufficient for Alpha. This item becomes active only when ChatHealthy exposes chatbot, API, Azure Function, or other compute-backed endpoints.

Objective: Add minimal server-side protections to prevent unauthorized access, abusive traffic, and uncontrolled compute cost once dynamic endpoints are live.

Scope:

  • /api/*, /chat, /search
  • Azure Function endpoints
  • Any future model, embedding, or workflow endpoints

Requirements:

  • Require API key on all protected endpoints via x-api-key header; reject missing or invalid keys with 401/403
  • Server-side rate limiting per IP or API key; return 429 when limits are exceeded
  • Public static content remains open — only dynamic endpoints are gated
  • Log rejected requests, auth failures, and rate-limit events
  • Default to deny on validation failure

Non-goals for first implementation: No OAuth, no user account system, no RBAC, no enterprise IAM integration.

Success criteria: No protected endpoint callable anonymously; excessive usage rate-limited; compute-backed endpoints cannot be abused without authorization; implementation compatible with solo-developer velocity.

Note: This is a cost-control and abuse-prevention measure, not a full enterprise security program. Implement immediately after Alpha once dynamic endpoints are exposed.

Bugs

BUG-001 Idle monitor can pause cluster while user is in Atlas GUI Low — dev inconvenience
Status: Open (idle monitor currently disabled)

The idle monitor fired every 30 minutes and paused ChatHealthyDataPipelines if no active load workers were detected and the last completed pipeline report was older than IDLE_MONITOR_THRESHOLD_HOURS (default 2h). It had no awareness of a human browsing the Atlas console or of a pipeline run in its early steps (before workers start).

Current state: Timer trigger is commented out in function_app.py. Cluster is managed manually.

Fix options: Re-enable only after adding a pipeline lock mechanism that the idle monitor respects. Alternatively, implement the cluster manager as a separate agent fully decoupled from the pipeline.
