1. Overview

The FindCare Chat Application is App 2 in ChatHealthy's three-application architecture. It handles all user interaction and real-time UX: LLM conversation, tool calls, and inline components. It reads from the database freely for fast UX lookups and may write to application collections (e.g. AboutUs). It must never write to PublicHealthData.

Stack

  • Frontend: React 18 + TypeScript, built with Vite, served as static files
    • Think timer & stop button
    • Rate limit auto-retry (dev only)
    • Emergency lock UI
    • Message rendering & markdown
  • Backend: Python FastAPI (Uvicorn), direct Anthropic SDK
  • Primary LLM: Claude Sonnet 4.6 (claude-sonnet-4-6)
  • Safety/Utility LLM: Claude Haiku 4.5 (claude-haiku-4-5-20251001)
  • Embeddings: OpenAI — see embedding model details
  • Database: MongoDB Atlas (ChatHealthyFrontEnd cluster)
  • Hosting: HuggingFace Docker Space (React static + FastAPI in single container)

Request Flow

The React frontend sends POST /chat with the user's message and full conversation history. The FastAPI backend runs a safety check, builds the system prompt, and calls Claude Sonnet 4.6 with the tool definitions. Claude selects tools, the backend executes them and returns results. Claude continues in a tool-use loop until it produces a final text response, which is returned to the frontend as a structured ChatResponse JSON object.

Supported States

Provider search is currently available in Delaware (DE) and Mississippi (MS) only. Requests for other states return a supported=false response with a note that interest has been recorded.

2. System Prompt

The system prompt is generated dynamically by _system_prompt(follow_up_check) and targets a budget of approximately 700 tokens. Context is loaded on demand through tool results, not pre-loaded unconditionally. This keeps per-request cost minimal for the majority of messages that do not require context documents.

Persona

The assistant is instructed to act as Skip Snow, answering questions on the ChatHealthy.AI website. It represents Skip Snow and ChatHealthy.AI for interactions as faithfully as possible — professional and engaging, as if talking to a potential client or future employer.

Rule Structure

The system prompt enforces behavior through a numbered RULE structure. Rules are labeled "STRICT ANSWER RULES — NO EXCEPTIONS" to maximize compliance.

Rule Name Behavior
RULE 0 EMERGENCY If the user describes ANY symptom or situation that could be a medical emergency, STOP immediately. Do not use any tool. Respond ONLY with the hardcoded emergency response text.
RULE 1 CONTEXT TOOLS Call get_skip_snow_context or get_chathealthy_context only when the user explicitly asks for information about Skip Snow or ChatHealthy. Do NOT call for greetings or casual conversation. Answer ONLY from what those tools return — never use general training knowledge.
RULE 2 NO HEDGING Prohibited from using any hedging language (I think, probably, might). If hedging words would be needed, call record_unknown_question instead.
RULE 3 MEDICAL ADVICE vs NAVIGATION Decline personal medical advice (call record_unknown_question first). Healthcare navigation is allowed: find_providers, find_specialty_codes, search_clinical_trials.
RULE 4 TOOL CALL ORDER Always call record_unknown_question BEFORE composing the response when the answer is not known.
RULE 5 EACH QUESTION SEPARATELY Record each unknown question with a separate tool call.
RULE 5b PROVIDER DISPLAY FORMAT MUST display provider results as a bullet list, never as a markdown table. Format: bold name, indented sub-bullets for address, county, phone (formatted (nnn) nnn-nnnn), NPI. Include a count line before results: "There are N [specialty] in [location]:"
RULE 5c EXTERNAL PROVIDER LOOKUP MUST call lookup_provider_external when the user asks about a specific provider by name.
RULE 6 FOLLOW-UP OFFER When a FOLLOW-UP CHECK reminder is appended, assess genuine interest and ask if the user would like someone from the team to follow up personally. Respect refusal signals.

Follow-Up Check Mechanism

Every 5th user message (user_msg_count % 5 == 0, starting after the first), the backend appends a FOLLOW-UP CHECK block to the system prompt. This instructs Claude to review the conversation and, if the user has shown genuine interest, offer to connect them with the team. The mechanism is invisible to the user — it is injected server-side only.

3. Router

The Router is a first-class architectural component of App 2. Its implementation is Claude's tool selector — not a separate service, not a workaround. Claude reads the user's message, selects the appropriate tool, and that selection IS the routing decision. No separate classification service or extra API call is required.

Design principle: The system prompt stays minimal (~700 tokens) for every request. Context loads on demand through tool results, not pre-loaded unconditionally.

4. Tool Protocol

Tools are defined in the anthropic_tools list and passed to every Claude API call. The backend executes tool calls via _handle_tool_calls(), which looks up the function by name in globals(), injects chat_history for tools that require it, and returns results as JSON-serialized tool_result blocks. Claude continues in a tool-use loop until stop_reason != "tool_use".

Tool-Use Loop

After each Claude API call, if stop_reason == "tool_use", the backend:

  • Extracts all tool_use blocks from response.content
  • Executes each tool function, injecting chat_history for record_user_details and record_unknown_question
  • Appends the assistant turn (with tool_use blocks) and a user turn (with tool_result blocks) to the message list
  • Calls Claude again with the updated messages
  • Accumulates input_tokens and output_tokens across all iterations

The loop continues until Claude produces a stop_reason other than "tool_use", at which point the final text block is extracted and returned.

Context Document Loading

Context documents are loaded once at startup by _load_me_context() into a module-level _ME dictionary. Source files include:

  • SkipSnowLinkedInProfile.pdf — parsed via pypdf
  • anthropic_principles.txt — ~500 tokens of operating principles
  • summary.txt — career summary
  • chatHealthy_ai_business_plan.pdf — parsed via pypdf
  • findcare-code-package.json — codebase metadata

Large documents are trimmed at query time by _trim(text, max_chars), which appends a truncation note when content exceeds the limit (3000 chars for LinkedIn and business plan).

6. Safety System

The safety system uses a dual-trigger OR logic design: either a keyword match OR an AI semantic check can independently trigger an emergency lock. This ensures emergencies are never missed even if one detection method fails.

Emergency Keyword List

Emergency detection uses a dual-path design: (1) keyword substring match and (2) Haiku semantic classifier. Either path triggering is sufficient to activate the emergency response (OR logic, fail-safe). This ensures emergencies are caught even if one method misses, while the semantic path adds intelligence beyond rigid keyword matching.

The AI prompt for the Haiku semantic classifier needs tuning prior to alpha release.

The EMERGENCY_KEYWORDS list contains 26 phrases covering major emergency categories. A case-insensitive substring match against the user's message triggers the keyword path. 

EMERGENCY_KEYWORDS = [
    "chest pain", "chest tightness", "heart attack",
    "can't breathe", "cannot breathe", "difficulty breathing",
    "trouble breathing", "not breathing", "stopped breathing",
    "stroke", "face drooping", "arm weakness", "sudden numbness",
    "severe bleeding", "bleeding out", "won't stop bleeding",
    "unconscious", "passed out", "unresponsive",
    "overdose", "took too many", "took too much",
    "suicide", "suicidal", "kill myself", "end my life",
    "seizure", "convulsing",
    "severe allergic reaction", "anaphylaxis", "throat closing",
    "choking",
]

Haiku Semantic Check

In parallel with keyword matching, the backend calls Claude Haiku 4.5 as a safety classifier. Haiku is given strict criteria:

  • EMERGENCY requires ALL of: (1) specific body location or system named, (2) acute onset or severity stated, (3) life-threat implied
  • PROBE for vague complaints: "I'm in pain" (location unknown), "I don't feel well", "something hurts"
  • Returns JSON: {"emergency": true|false, "confidence": 0.0-1.0}
  • AI hit triggers only when emergency == true AND confidence >= 0.80

Dual-Trigger OR Logic

The final safety decision in _safety_check() is: return keyword_hit or ai_hit. If either path fires, the emergency protocol activates. If the Haiku call fails for any reason (network, parse error), the system defaults to escalation (return True) — fail-safe behavior.

Emergency Response

When triggered, the backend returns a hardcoded response:

<b>Call 911 or go to the nearest emergency room immediately. Do not wait.</b>

<b>This chat has been suspended.</b>

The response includes emergency: true in the JSON, which causes the frontend to set isLocked = true, disabling the input field.

IP Emergency Lock

  • In-memory: _ip_emergency_locks dict maps IP to expiry timestamp
  • Lock duration: 3600 seconds (1 hour)
  • Check order: in-memory first, then DB fallback via _check_ip_lock_db()
  • Client IP: Uses X-Forwarded-For header to obtain the real client IP behind proxies; falls back to request.client.host if the header is absent
  • DB upsert: One record per IP (upsert on ip field), not duplicates. Repeated emergencies from the same IP update the existing record.
  • Session detection: _session_is_locked() scans history for the emergency response text
  • Admin unlock: An administrator can unlock a locked session by sending the unlock key directly (no UNLOCK: prefix required). Comparison is case-insensitive. The unlock key is stored in the secrets repository as ADMIN_UNLOCK_KEY and is not exposed to the client.

DB Persistence

Emergency incidents are persisted to {env}_Safety.emergency_incidents via upsert on the ip field. One record per IP; repeated emergencies update the existing record rather than creating duplicates. 

{
    "ip": "...",
    "locked_at": "ISO 8601",
    "expires_at": "ISO 8601",
    "trigger_message": "(first 500 chars)",
    "chat_history_deidentified": [
        {"role": "user|assistant", "content": "(first 300 chars)"}
    ]
}

If MongoDB is unavailable at lock time, the in-memory lock still activates and a warning is logged. The incident is not persisted but the user is still protected.

Admin Unlock

An admin can unlock a locked IP by sending the unlock key directly as the message (no UNLOCK: prefix required). Comparison is case-insensitive. This clears the in-memory lock and marks the DB record as unlocked rather than deleting it, preserving the full audit trail: 

{
    "unlocked": true,
    "unlocked_at": "ISO 8601",
    "unlocked_by": "admin"
}

The DB check (_check_ip_lock_db()) skips unlocked records by filtering with {"unlocked": {"$ne": true}}. The unlock key is set via the ADMIN_UNLOCK_KEY environment variable.

BTC-003 — MUST HAVE: The safety audit trail is a must-have requirement. Emergency incident records must never be deleted. Unlock marks the record as unlocked but preserves the original lock data for compliance review.

7. Provider Search

Provider search follows a cascading strategy: vector search primary, taxonomy fallback, county physician fallback. Each layer is tried in sequence, and the first to return results is used.

Search Path

  1. Vector search — embed the specialty query via OpenAI text-embedding-3-large, run $vectorSearch against provider_vector_index with state pre-filter, then apply city/county post-filters
  2. Taxonomy code fallback — if vector search returns 0 results or embedding fails, call find_specialty_codes() to resolve NUCC taxonomy codes, then query providers by taxonomies.code
  3. County physician fallback — if taxonomy search also returns 0 results AND a county was specified, broaden to all individual providers (entity_type_code=1) whose primary taxonomy starts with ^2 (NUCC physician block) in that county

Vector Search Details

  • Index: provider_vector_index on providers collection
  • Path: embedding field
  • numCandidates: min(limit * 30, 300)
  • limit: min(limit * 6, 60)
  • Pre-filter: practice_address.state (required, passed in $vectorSearch.filter)
  • Post-filters: city (regex, case-insensitive), county (via _make_county_filter)
  • Hard limit: capped at 10 results (safe_limit = min(int(limit), 10))

Specialty Code Resolution

find_specialty_codes() runs two parallel pipelines using ThreadPoolExecutor(max_workers=2):

  • Regex pipeline: Haiku expands the query into 2-5 keyword stems, then regex-searches Specialization and Display Name fields in SpecialtyMetaData
  • Vector pipeline: Embeds query via text-embedding-3-small, runs $vectorSearch against specialty_vector_index, filters results above score 0.4, and pulls all codes in matching classifications

Results are deduplicated by code, vector results prioritized, capped at 70. Only individual provider groupings are included (16 NUCC groupings covering physicians, nurses, therapists, etc. — excluding facilities, agencies, and group practices).

County Filter: _make_county_filter

The _make_county_filter(county) function builds a MongoDB filter that matches by county name OR by FIPS code. This is necessary because of bug ASN-4AFBDA: 2,579 providers have county.fips set but county.name blank. 

def _make_county_filter(county: str) -> dict:
    term = county.strip().lower()
    name_filter = {"county.name": {"$regex": county.strip(), "$options": "i"}}
    matching_fips = [fips for fips, name in _fips_to_county.items()
                     if term in name.lower()]
    if matching_fips:
        return {"$or": [name_filter, {"county.fips": {"$in": matching_fips}}]}
    return name_filter

FIPS Hack (ASN-4AFBDA)

The _fips_to_county mapping is pre-seeded with a static table of all 85 counties in DE (3) and MS (82), then augmented at startup from the providers collection itself via _load_fips_county_map(). DB values override the static seed. This ensures county search works even for the worst case where ALL records in a county lack county.name.

Provider Formatting

_format_provider() normalizes provider records for display:

  • Individual providers (entity_type_code=1): assembles name from prefix, first, middle, last, suffix, and credential text
  • Organizations: uses provider_organization_name_legal_business_name
  • Address: joins line1, city, state, zip from practice_address
  • County: uses county.name, falling back to FIPS lookup via _fips_to_county
  • Taxonomy code: the primary taxonomy code from the taxonomies array

8. UI/UX Behavior

ChatWindow State Machine

The ChatWindow component manages six state variables:

State Type Purpose
messages Message[] Full conversation history (user + assistant messages). Initialized with WELCOME.
input string Current text in the input field.
isLoading boolean True while awaiting a backend response. Controls the thinking indicator and disables send (unless dismissed).
isLocked boolean True after an emergency response. Changes placeholder to "Chat suspended. Type UNLOCK code to resume." Input stays enabled for unlock key entry. Cleared when backend responds with "Session unlocked."
thinkSeconds number Counter incremented every second while loading. Reset to 0 when loading ends or thinking is dismissed.
thinkingDismissed boolean True if the user clicked "Stop". Hides thinking indicator, re-enables send, but does not cancel the backend request.
retryCountdown number | null Counts down from 10 seconds during rate limit auto-retry. Null when not in retry state.

Message Interface

interface Message {
  role: 'user' | 'assistant'
  content: string
  isError?: boolean
  thinkSeconds?: number
  tokensIn?: number
  build?: string
}

Submit Logic

canSubmit = !isLoading || thinkingDismissed

Submit is allowed when the backend is idle OR the user has dismissed the thinking indicator. The isLocked state does not block submit — the user must be able to type and send the unlock key while locked.

Think Timer and Stop Button

↑ Back to overview

  • The thinking indicator shows "Thinking... Ns" with a "Stop" button
  • The timer counts up every second while isLoading && !thinkingDismissed && retryCountdown === null
  • Clicking "Stop" sets thinkingDismissed = true, which hides the indicator and re-enables the send button
  • "Stop" does NOT cancel the in-flight request — the response will still arrive and be appended to the conversation
  • Both thinkingDismissed and thinkSeconds reset when the response arrives

Rate Limit Auto-Retry (Dev Only)

↑ Back to overview

When the backend returns error_type: "rate_limit" AND the backend environment is "dev" (determined from /health response):

  • The pending message and history are saved to pendingRetryRef
  • A 10-second countdown is displayed: "Rate limit — retrying in Ns" (orange styling)
  • When countdown reaches 0, the original request is automatically retried via doApiCall()
  • isLoading remains true throughout — the thinking indicator stays visible
  • In production, rate limit errors display as normal error messages with no auto-retry

9. Error Handling

ChatResponse Model

class ChatResponse(BaseModel):
    response: Optional[str] = None
    emergency: bool = False
    error: Optional[str] = None
    error_type: Optional[str] = None   # rate_limit | db_unavailable | upstream | internal
    tokens_in: Optional[int] = None
    tokens_out: Optional[int] = None

Error Types

error_type Trigger Frontend Behavior
rate_limit Exception string contains "429" or "rate_limit" Dev: 10-second countdown + auto-retry. Prod: displayed as error message.
db_unavailable Exception string contains "unavailable" or "connection" Displayed as error message with code block.
upstream External service failures (ClinicalTrials.gov, embedding API) Error returned inside tool result; Claude explains it to the user.
internal All other unhandled exceptions Full traceback shown in dev (DEBUG=true); short error string in prod.

Backend Error Flow

The outer chat() endpoint wraps _chat_inner() in a try/except. Any unhandled exception is caught, classified by error type, logged to dev_Debug.chat_calls (dev only), and returned as a ChatResponse with the error and error_type fields populated. On error, the full de-identified chat history is stored alongside the error record for debugging. History is truncated to 500 chars per message and run through deIdentify() before persistence.

Frontend Error Display

  • If data.error is present, the error text is wrapped in a Markdown code block: **Error:**\n```\n{error}\n```
  • The message is marked with isError: true, which triggers red styling in MessageBubble
  • If the fetch itself fails (network error), a hardcoded error message is shown: "Could not reach the server. Please try again."

Debug Logging

When ENV_PREFIX == "dev", every chat call is logged to dev_Debug.chat_calls with: datetime, IP, message preview (200 chars), history length, tool loop iterations, token counts, response preview (200 chars), and error text if any.
