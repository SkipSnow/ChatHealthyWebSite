1. Product Vision

ChatHealthy / FindCare is a unified healthcare discovery platform that turns fragmented public provider data into a living, queryable knowledge system. A continuously updated data pipeline ingests national provider datasets, normalizes and enriches them with geographic intelligence (ZIP → county → state), and organizes the result into structured collections and vectorized knowledge layers.

On top of this foundation, domain-specific AI agents guide users to meaningful healthcare queries — finding specialists by geography, understanding provider distribution, exploring care options — with precise, quantitative, and explainable answers. Users engage through a dynamic interface where healthcare data is visual, navigable, and actionable, supporting everything from high-level regional insights to individual care decisions.

The goal is to democratize healthcare intelligence — making complex provider ecosystems understandable and useful for real-world decision-making.

Non-Modal Slide-In Panel

Pages open inside a slide-in panel anchored to the right edge of the viewport. The chat iframe remains alive underneath — the panel is not modal and does not block interaction. Each page detects whether it is loaded inside the panel (via window !== window.top) and adds an in-panel class to <body>, which reveals Close buttons and hides the page header.

7. Data and Privacy

HIPAA Consent Flow

Q1: "May we save a verbatim transcript of this conversation with your contact details?"
Yes → Store full chat history (PII intact, consent exchange included as evidence)
Q2 (if Q1 declined): "May we save a de-identified summary instead?"
Yes → LLM summarizes → deIdentify() scrubs PII → stored in notes field only
Neither consented → Contact fields only; no history stored
Unknown questions always de-identified and recorded regardless of consent

9. Data Pipeline Architecture

Public health data (providers, specialties, ICD-10, procedures, prescriptions) is loaded and enriched by App 3 on a monthly cadence. The pipeline is triggered manually via POST to /api/Router. The MongoDB Atlas cluster (ChatHealthyDataPipelines) is started manually before each run. A health check at pipeline start pings the cluster, emails the admin, and fails fast if it is unreachable — no workers are launched against a down cluster.

FullProviderPipeline — Current Implementation (5 Steps)

All steps run sequentially. Pass start_step (1–5) in the payload to resume from any step without re-running prior completed work. Default: full run from Step 1.

12. Deferred

  • CopyToRuntime + Blue-Green Collection Swap — fan-out copy of providers_staging → providers_new on ChatHealthyFrontEnd cluster, then atomic rename swap. Until implemented, runtime reads are served from the DataPipelines cluster.
  • Cluster manager agent — dedicated agent/service to start/stop the DataPipelines cluster on a schedule, fully decoupled from the pipeline itself. Current approach: manual.
  • Idle monitor re-enable — auto-pause the DataPipelines cluster when idle. Disabled because it fired mid-pipeline run. Requires a pipeline lock mechanism before re-enabling.
  • ICD-10 → specialty → provider research agent — given a diagnosis code, identify relevant specialties and surface providers + prescribing patterns (see RESEARCH-002).
  • Multi-location provider aggregation — NPPES has one practice address per individual NPI. Supplementing with payer directory data would reveal all locations where a provider bills (see RESEARCH-001).
  • FHIR / EMR integration — Layer 2, future phase
  • User authentication and cross-session persistence
  • Production MongoDB cluster (currently dev only)
  • MongoDB connection pooling design
  • Data deletion agent — Azure Function with email verification, MongoDB deletion, confirmation email
  • Migrate Layer 1 off HuggingFace → Azure Static Web Apps. HuggingFace has a 60-second request timeout (too short for multi-agent workflows), spaces sleep on inactivity, and free-tier compute is undersized for production. Azure Static Web Apps places the React frontend on Azure CDN and the FastAPI backend as an Azure Function — same infrastructure already running DataPipelines, single vendor, single deploy pipeline. HuggingFace stays live until Layer 1 is replaced.
  • Enterprise security hardening. Current setup uses workstation-cached GitHub credentials and manually managed secrets in Azure portal — acceptable for solo dev, insufficient for enterprise customers. Pre-sales blockers: fine-grained GitHub PAT or SSH key with passphrase; Azure Key Vault for secrets management; GitHub Actions OIDC federation replacing long-lived secrets; GPG-signed commits for verifiable audit trail; separation of deploy and dev credentials.
