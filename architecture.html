1. Vision and Mission

Vision: ChatHealthy.ai provides the data, tools, and an AI-enlightened space to empower regular people to prevail over the broken healthcare system. Whatever a person's circumstances might be, they deserve the best possible care, at the best value. ChatHealthy.ai gives them the platform to get it.

Mission: Help people make better healthcare decisions together.

ChatHealthy is not cloud-native — it is AI-native. Every architectural decision, from data pipeline design to the conversational interface, is made with AI as the primary runtime, not an add-on.

Core Principle: "The model may suggest. The system must decide." — Token prediction is not truth. All AI-generated output is probabilistic language, not verified fact. No system behavior may treat model output as authoritative truth.

3. Governance Constraints (GOV-001 – GOV-012)

Governance policies are permanent unless explicitly marked temporary. They override convenience, speed, and model capability. Enforcement is mandatory before production release.

IDNamePolicy SummaryStatus
GOV-001 Provider Neutrality No revenue from health plans, insurance companies, PBMs, providers, or pharma. Revenue limited to consumer subscriptions and technology vendor partnerships. Neutrality is a core product principle. Permanent
GOV-002 Single-Tenant Architecture Single-tenant by design. Multi-tenancy is a healthcare anti-pattern. Each deployment has isolated data. Permanent
GOV-003 Public By Default All code, architecture, and design docs are public. Nothing is marked confidential. Copyright © Skip Snow. Temporary
GOV-004 HIPAA Consent Framework Two-tier consent flow: (1) verbatim transcript, (2) de-identified summary, (3) contact-only if both declined. Safety audit trail stored without consent as operational record. Permanent
GOV-005 Four-Application Boundary Exactly four applications in two categories. Product: Static Website, Chat Window, Data Pipelines. DevOps: The Brain. Integration only via prescribed patterns. No application may bypass boundaries. Permanent
GOV-006 Real System Testing End-to-end tests use real systems (MongoDB, Anthropic, OpenAI). No mocks in e2e. Test data tagged testdata=true, persisted in dev database (no teardown). Permanent
GOV-007 Deploy Authorization Dev deploys freely. Production requires explicit human sign-off. Staging authorized by GPT (Architect). QA-to-prod authorized by human only. No workarounds. Permanent
GOV-008 Business Artifacts Gate No production release without: QA Report PDF with GPT release gate approval, IP License Agreement, Business Model. These are business documents, not engineering artifacts. Permanent
GOV-009 Role In System Prompt Every AI agent role definition must be in the system prompt, never in the user prompt. System prompt is authoritative for agent identity. User prompt contains only the assignment payload. Permanent
GOV-010 Zero Data Exposure Chat interface must NEVER expose stored user data regardless of prompt phrasing. No record counts, no aggregations, no transcripts. Adversarial extraction tests required on every release. Permanent
GOV-011 AI Translates, System Answers "Only facts can answer a user question." AI turns user language into structured signals. The system answers using real data, not guesses. System-generated content takes precedence. User-facing output uses the user's own language, not professional terminology. Permanent
GOV-012 Environment Isolation Data moves only within its environment chain: dev → staging → prod. No cross-environment copies. No backward migration. Only human can authorize deviation. Permanent

GOV-011 Implementation Standards

StandardRuleEnforcement
GOV-011-STD-001
System Content Precedence		 When both the system and the AI produce output, system-generated content takes precedence. Redundant AI content is stripped. System summary, counts, links, and metadata are authoritative. A small model (Haiku) strips redundant summary/pagination language from AI response when a system summary_message exists.
GOV-011-STD-002
User Terms Over Professional Terms		 All user-facing output uses the user's own language. If the user says "shrinks", the system says "shrinks" — not "psychiatrists". AI may translate internally for query resolution. summary_message built from user's original message (body.message), not from the LLM's translated tool parameters.

5. Integration Patterns

Application boundaries are enforced by GOV-005. Only three integration patterns are permitted:

PatternFrom → ToMechanismPurpose
iframe embed App 1 → App 2 HTML <iframe> element on the static website loads the Chat Window Decouple frontend presentation from application logic. Website is pure static HTML with zero backend.
HTTP POST /api/Router App 2 → App 3 Authenticated HTTP POST with task payload. Returns 202 Accepted immediately. Work executes asynchronously via Azure Durable Functions. Trigger pipeline operations (provider load, enrichment, embedding) from the chat application or administrative tooling.
GPTReader API App 4 (Brain) → App 3 Read-only Azure Function endpoint. Authenticated, audited, 27/27 SIT pass. Returns brain context as JSON. Allow the AI-native SDLC (Brain) to read project context through the pipeline infrastructure without direct database access.

Prohibited Patterns

  • Direct database access across application boundaries
  • Shared in-process state between applications
  • Any integration pattern not listed above

6. Data Flow — AI Translates, System Answers (GOV-011)

This is the fundamental data flow for every user interaction. The AI's role is translation (natural language to structured intent). The system's role is resolution (query real data, apply rules, return bounded results). The AI must never generate answers from its own knowledge when system data exists.

User Message"Find me a cardiologist in Delaware"
↓ AI translates natural language to structured intent
AI Translation LayerClaude extracts: specialty="cardiologist" · state="DE"
Selects tool: find_providers
↓ structured parameters passed to system
System Resolution LayerMongoDB query: SpecialtyMetaData → taxonomy codes → providers collection
Deterministic · auditable · no AI involvement
↓ real data returned
System-Built Summarysummary_message built from user's original words + real data counts
GOV-011-STD-002: "shrinks" not "psychiatrists"
↓ system content takes precedence (GOV-011-STD-001)
Response to UserSystem summary + provider cards + inline components
AI adds only non-redundant conversational context

System Roles (AI Governance Policy)

LayerRoleConstraints
AI Model Translate user language into structured signals. Suggest possible mappings. Generate explanations. Must NOT make final decisions. Must NOT introduce facts outside system data. Must NOT be treated as source of truth.
Data Layer Single source of truth. All factual, auditable information from verified public or consented sources. All outputs must trace back here. No model-generated data may overwrite or augment it.
Decision Layer Apply rules, mappings, and scoring. Determine bounded outputs. Enforce constraints and guardrails. Must be deterministic or explainable. Must be auditable. Must NOT defer authority to the model.
Explanation Layer Present results in clear, user-friendly language. Describe signals and reasoning used. Must reflect actual system logic. Must NOT fabricate justification. Must NOT imply certainty where none exists.

Final statement: "Fluency is not evidence. Models assist. Systems decide. Data grounds truth."

9. Security Posture

ChatHealthy.ai is NOT a HIPAA covered entity and is NOT subject to HIPAA regulations. We are a consumer navigation platform, not a healthcare provider, business associate, or regulated medical device. We voluntarily adopt enterprise-grade security standards because our business depends on consumer trust, not regulatory compulsion.

Every control required by NIST SP 800-171 (CUI protection), HITRUST CSF (healthcare security), and SOC 2 Type II (operational security) is explicitly mapped to our engineering operating procedures and embodied in our deployed logical and physical architecture. These are not aspirational targets — they are implemented controls with documented evidence. From a technology architecture perspective, we believe we are fully SOC 2 compliant today. The formal audit is a matter of scheduling, not remediation.

"We are not required to protect your data. We choose to. And we do it better than the ones who are required to."

Voluntary Security Certification Path

StandardWhatWhy
HITRUST CSF Healthcare-specific security framework. Gold standard for health tech. Maps to HIPAA, NIST, ISO 27001. Healthcare buyers require it. Proves we exceed HIPAA without being subject to it.
SOC 2 Type II Independent audit of security controls over 6–12 months. Trust Services Criteria. Enterprise table stakes. Proves operational security, not just policy.
NIST SP 800-171 DOD standard for Controlled Unclassified Information. 110 security controls. Direct path to FedRAMP. If VA or DOD wants ChatHealthy, we are ready.
Annual Pen Test Third-party security audit. Published results. Transparency. We publish what hospitals hide.

Security Disclosure

Every session and every document discloses: (1) We are not subject to HIPAA. (2) We voluntarily maintain audited security at enterprise healthcare standards. (3) Even if a participant happens to be a licensed provider, the platform remains a consumer tool and does not create a provider-patient relationship.

10. Data and Privacy

HIPAA Consent Flow (GOV-004)

Q1: "May we save a verbatim transcript of this conversation with your contact details?"
Yes → Store full chat history (PII intact, consent exchange included as evidence)
Q2 (if Q1 declined): "May we save a de-identified summary instead?"
Yes → LLM summarizes → deIdentify() scrubs PII → stored in notes field only
Neither consented → Contact fields only; no history stored
Unknown questions always de-identified and recorded regardless of consent
