1. Product Vision

ChatHealthy / FindCare is a unified healthcare discovery platform that turns fragmented public data into an AI based knowledge system. A continuously updated data pipeline ingests national provider datasets, normalizes and enriches them with geographic intelligence (ZIP → county → state), and organizes the result into structured collections and vectorized knowledge layers.

On top of this foundation, domain-specific AI agents guide users to meaningful healthcare queries — finding specialists by geography, understanding provider distribution, exploring care options — with precise, quantitative, and explainable answers. Users engage through a dynamic interface where healthcare data is visual, navigable, and actionable, supporting everything from high-level regional insights to individual care decisions.

The goal is to democratize healthcare intelligence — making complex provider ecosystems understandable and useful for real-world decision-making.

ChatHealthy is not cloud-native — it is AI-native. Every architectural decision, from data pipeline design to the conversational interface, is made with AI as the primary runtime, not an add-on.

Pages open inside a slide-in panel anchored to the right edge of the viewport. The chat iframe remains alive underneath — the panel is not modal and does not block interaction. Safety System LIVE — MANDATED

Safety detection is a hard system invariant. It executes before every message — before any model call, before any tool. False positives are acceptable; false negatives are not. Policy authority: Skip Snow.

User Message
↓ first — before everything
Dual Safety DetectionKeyword match (rule-based) + Haiku semantic check — EITHER signal triggers escalation
Emergency signal
Hard Stop — IP locked 1 hourFixed response only · no model call · no tools · all subsequent messages locked
Safe
Normal flowIntent routing → tool calls → response

Fixed response (hardcoded — no variation allowed): "This may be a medical emergency. Call 911 or go to the nearest emergency room immediately. Do not wait."

Failure default: if safety detection errors or times out, the system escalates — it does not continue.

IP Emergency Lock

When a safety trigger fires, the source IP is locked for one hour. Every subsequent message from that IP returns the fixed emergency response regardless of content. The lock is held in process memory and does not survive a container restart; DB persistence is planned.

An admin backdoor clears the lock. The backdoor key is stored in the ADMIN_UNLOCK_KEY environment variable. The unlock command is the exact string UNLOCK:<key> (case-insensitive, must be the entire message). The key value is not documented here; it is set in deployment environment secrets.

10. Data Pipeline Architecture

Public health data (providers, specialties, ICD-10, procedures, prescriptions) is loaded and enriched by App 3 on a monthly cadence. The pipeline is triggered manually via POST to /api/Router. The MongoDB Atlas cluster (ChatHealthyDataPipelines) is started manually before each run. A health check at pipeline start pings the cluster, emails the admin, and fails fast if it is unreachable — no workers are launched against a down cluster.

FullProviderPipeline — Current Implementation (8 Steps)

All steps run sequentially. Pass start_step (1–8) in the payload to resume from any step without re-running prior completed work. Default: full run from Step 1.

Multi-Pass Enrichment Pattern

Pipeline enrichment applies three escalating tiers. Records that cannot be resolved by a cheaper tier advance to the next. Only the minimum necessary tier runs for each record.

TierMethodCost
1 — DataBulk resolution from authoritative structured datasets — ZIP crosswalk, Census Geocoder, address fallbacksLowest
2 — WebTargeted web search by NPI, then by provider name + stateMedium
3 — AIResearch agent — reasons across multiple sources for records data and web cannot resolveHighest

County enrichment is the first pipeline to implement this pattern. The same three-tier approach applies to any enrichment concern across the platform.

ICD-10 → Specialty → Provider Research Agent ROADMAP — MVP

Not yet implemented. Pipeline agent that enriches the knowledge base with ICD-10 → specialty → prescribing pattern relationships.

Given a diagnosis code (ICD-10-CM), the research agent identifies relevant medical specialties, surfaces providers who treat that condition, and analyzes prescribing patterns from CMS Part D data. This enriched knowledge layer powers the FindCare symptom → care options path.

Pipeline Exception Handling Framework

All pipeline row-processing workers extend PipelineWorkerBase, a Gang of Four Template Method base class. The base class owns the processing loop, all exception catching and logging, and the row_errors accumulator. Worker subclasses implement data processing logic only — they do not catch exceptions or log errors. A fail_on_row_error flag (default False) controls whether a single bad row abends the job or is logged, recorded, and skipped. See the Provider Data Load design doc §4b for full detail.

Two-Cluster Architecture (planned)

Pipeline and runtime workloads will be separated into two Atlas clusters to prevent bulk write operations from impacting app query performance. Collection swap and CopyToRuntime are deferred (see Section 12):

ClusterPurpose
ChatHealthyFrontEndAlways on. App reads (providers, specialties, chat tools) + all admin data: prod_Brain (brain loop), prod_Backlog (product backlog), AboutUs (leads, unknown questions).
ChatHealthyDataPipelinesStarted manually before each pipeline run, stopped when done. Batch writes only — provider load, enrichment, embeddings. Never holds admin or operational data.

13. Deferred

  • CopyToRuntime + Blue-Green Collection Swap — fan-out copy of providers → providers_new on ChatHealthyFrontEnd cluster, then atomic rename swap. Until implemented, runtime reads are served from the DataPipelines cluster.
  • Cluster manager agent — dedicated agent/service to start/stop the DataPipelines cluster on a schedule, fully decoupled from the pipeline itself. Current approach: manual.
  • Idle monitor re-enable — auto-pause the DataPipelines cluster when idle. Disabled because it fired mid-pipeline run. Requires a pipeline lock mechanism before re-enabling.
  • Multi-location provider aggregation — NPPES has one practice address per individual NPI. Supplementing with payer directory data would reveal all locations where a provider bills (see RESEARCH-001).
  • FHIR / EMR integration — Layer 2, future phase
  • User authentication and cross-session persistence
  • Production MongoDB cluster (currently dev only)
  • MongoDB connection pooling design
  • Data deletion agent — Azure Function with email verification, MongoDB deletion, confirmation email
  • Migrate Layer 1 off HuggingFace → Azure Static Web Apps. HuggingFace has a 60-second request timeout (too short for multi-agent workflows), spaces sleep on inactivity, and free-tier compute is undersized for production. Azure Static Web Apps places the React frontend on Azure CDN and the FastAPI backend as an Azure Function — same infrastructure already running DataPipelines, single vendor, single deploy pipeline. HuggingFace stays live until Layer 1 is replaced.
  • Enterprise security hardening. Current setup uses workstation-cached GitHub credentials and manually managed secrets in Azure portal — acceptable for solo dev, insufficient for enterprise customers. Pre-sales blockers: fine-grained GitHub PAT or SSH key with passphrase; Azure Key Vault for secrets management; GitHub Actions OIDC federation replacing long-lived secrets; GPG-signed commits for verifiable audit trail; separation of deploy and dev credentials.
